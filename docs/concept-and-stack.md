# 产品构想与细节技术栈

## 1. 产品构想收敛

### 一句话定位

为中小企业提供可私有化部署的 AI 会议基础设施，解决会议过程留痕、会后纪要、历史检索和数据主权问题。

### 不做什么

这个产品不以“功能最多的视频会议软件”为目标，也不以“炫酷 AI 演示”作为首发卖点。

首发阶段不追求：

- 替代所有主流会议软件的全部功能
- 做通用协同操作系统
- 做情绪识别、手势白板、音色模拟一类高研发成本功能

### 首发切口

最适合的切口是：

- 企业内部例会、项目评审、客户复盘
- 对录音录像归档和纪要质量有要求
- 对“数据不出域”敏感
- 已经有 OA、IM、知识库，希望会议资产能沉淀进去

### 用户真实购买理由

用户购买不是因为“界面更炫”，而是因为下面四件事能直接提升管理效率：

- 所有会议资产都在自己控制的环境里
- 会后几分钟能拿到可用纪要和待办
- 历史会议可按人、按主题、按时间检索
- 能接入现有身份、日程、存储和归档体系

## 2. v1 核心业务流

### 会前

- 管理员部署系统并绑定对象存储或 NAS
- 配置组织、成员和会议权限
- 外部 OA、IM 或日历通过 webhook / API 创建会议

### 会中

- 用户通过 Web 客户端进入会议
- 系统通过 RTC 服务承载音视频、屏幕共享和录制
- 应用服务记录会议元数据和访问审计

### 会后

- 录制完成事件触发会后处理流水线
- 流水线执行转写、摘要、待办提取和检索索引构建
- 最终结果写入客户指定存储，并通过搜索和回放界面暴露

## 3. 技术栈决策

## 前端

- `Next.js 15 + React 19 + TypeScript`
- `Tailwind CSS + shadcn/ui`
- `TanStack Query`
- `LiveKit Client SDK`

选择理由：

- Next.js 适合同时承载管理后台、会议入口、回放与纪要检索页面
- React 生态和 LiveKit 集成成熟
- TypeScript 便于与控制面接口保持一致

## 控制面后端

- `Go 1.23+`
- `Gin`
- `pgx`
- `sqlc`
- `goose`

选择理由：

- Go 更适合承接会议控制面、Webhook、权限、存储适配和异步任务编排
- 单二进制交付对私有化部署更友好
- `pgx + sqlc` 比 ORM 更容易保持 SQL 可控性和性能边界
- 对小团队来说，Go 的开发和运维成本通常低于 Rust，同时明显更贴近这类 I/O 型系统需求

## 异步任务与会后流水线

- `Asynq`
- `Redis`
- `FFmpeg`

职责：

- 录制完成后的任务调度
- 音频抽取、转写、摘要、待办提取
- 存储写入、失败重试、状态回传

选择理由：

- Asynq 在 Go 生态里足够成熟，适合会后异步任务和失败重试
- Redis 既可做 broker，也可做短期状态缓存
- 会后处理天然适合异步流水线

## 实时音视频

- `LiveKit` 自部署版

选择理由：

- 支持多人会议、屏幕共享、录制和房间管理
- Web SDK 成熟，私有化部署路径清晰
- 比从零构建 WebRTC 控制层更省团队资源

不建议首发自研：

- SFU
- 录制基础设施
- 复杂设备自适应策略

## 数据层

- `PostgreSQL 16`
- `Redis`
- `S3-compatible Object Storage`，首推 `MinIO`

PostgreSQL 职责：

- 组织、成员、角色
- 会议元数据
- 任务状态
- 转写片段索引
- 纪要与待办结构化结果

Redis 职责：

- Asynq broker
- 短期任务状态缓存
- 限流与会话辅助数据

对象存储职责：

- 录音
- 录像
- 转写原文
- 导出纪要

NAS 支持方式：

- 通过挂载目录接入
- 后端统一以“存储适配器”方式读写，避免业务代码直接依赖底层存储形式

## 搜索

v1 选择：

- `PostgreSQL Full-Text Search`

原因：

- 中小企业首发规模下足够
- 运维成本显著低于 OpenSearch
- 可与会议元数据、权限过滤一起处理

暂不作为 v1 强依赖：

- 向量数据库
- 复杂 RAG 检索链

## AI 能力

### 语音转写

优先方案：

- `FunASR / SenseVoice`

备选方案：

- `faster-whisper`

选择理由：

- 中文场景表现更稳
- 支持本地化部署
- 更容易满足“数据不出域”

### 纪要与待办提取

优先方案：

- `vLLM` 部署 `Qwen2.5-Instruct` 系列模型

调用方式：

- 对内统一为 OpenAI-compatible API，由 Go 服务通过 HTTP 调用

选择理由：

- 中文会议场景质量与成本平衡更好
- 模型服务可独立部署，不把 LLM 推理逻辑嵌进主后端

## 身份与权限

v1 默认能力：

- 本地账号体系
- RBAC 角色控制

企业集成预留：

- `OIDC`
- `SAML`
- `LDAP/AD`

实现建议：

- 先把内部身份模型和角色模型做清楚
- 外部 IdP 通过适配器接入
- 不在 v1 强绑定某个身份中台产品

## 部署与运维

试点阶段：

- `Docker Compose`
- `Nginx` 作为反向代理

规模化阶段：

- `Kubernetes + Helm`

首发不建议直接上 Kubernetes 的原因：

- 目标客户是中小企业，不是大规模多集群 SaaS
- 试点阶段需要的是快速部署和低运维门槛

## 监控与审计

- `Prometheus`
- `Grafana`
- `Loki`
- `OpenTelemetry`

核心监控项：

- 会议在线状态
- RTC 房间状态
- 录制完成率
- 转写任务时延
- 存储写入失败率
- 回放与纪要访问审计

## 4. 推荐代码仓结构

```text
ai-meeting/
  apps/
    web/
  services/
    api/
    jobs/
  packages/
    contracts/
  infra/
    docker/
    helm/
  docs/
```

模块职责：

- `apps/web`：会议入口、管理后台、回放与检索 UI
- `services/api`：组织、会议、权限、资产访问、外部接口
- `services/jobs`：异步任务消费、转写编排、摘要编排、归档
- `packages/contracts`：前后端共享的数据契约
- `infra`：私有化部署模板和环境配置

## 5. 明确的技术取舍

做：

- 自部署 LiveKit
- Go 承担控制面和异步编排层
- PostgreSQL 先承担结构化数据和全文检索
- 对象存储和 NAS 统一为存储适配器
- Docker Compose 作为首发部署形态
- ASR 和 LLM 作为独立模型服务，通过 HTTP API 接入

不做：

- 自研 RTC
- 一开始全仓采用 Rust
- 一开始就引入向量数据库
- 一开始就做 Electron 客户端
- 一开始就做复杂 Agent 编排能力

## 6. Go 与 Rust 的取舍结论

当前阶段优先选择 Go，原因是：

- 项目核心难点是系统集成、部署交付、权限和异步编排，不是极限性能
- Go 在私有化交付、并发服务、运维和团队扩展上更均衡
- Rust 更适合后续可能出现的热点模块，例如高性能媒体处理、边缘代理或协议层组件

建议策略：

- 主后端全部使用 Go
- ASR、LLM 独立为模型服务
- 若后续出现明确性能热点，再局部引入 Rust，而不是一开始全局采用 Rust
